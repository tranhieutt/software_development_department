Kỹ thuật 01
Hook Orchestration
6 event types — mỗi event có nhiều handler tạo vòng kiểm soát tự động quanh mọi hành động của agent.
Event Hook File(s) Harness Pattern
SessionStart session-start.sh
detect-gaps.sh		 Load sprint context, git activity; phát hiện project mới, docs thiếu; recover session trước từ active.md
PreToolUse (Bash) bash-guard.sh
validate-commit.sh
validate-push.sh		 Hard-block lệnh nguy hiểm; 4-layer commit quality gate; kiểm tra protected branch trước push
PreToolUse (Write/Edit) pre-refactor-impact.sh Nhắc chạy /gitnexus-impact-analysis trước khi sửa bất kỳ file trong src/
PostToolUse (Write/Edit) validate-assets.sh Kiểm tra naming convention và JSON validity cho file trong assets/
PreCompact pre-compact.sh Dump toàn bộ session state (active.md, modified files, WIP docs) vào conversation trước khi context bị nén
Stop session-stop.sh Archive session summary → production/session-logs/; kiểm tra 3 điều kiện auto-dream; trigger nếu thỏa
SubagentStart log-agent.sh Ghi JSONL audit log cho mọi lần spawn subagent — không agent nào có thể khởi động mà không bị log
Core Pattern: User action → Event fired → Hook intercepts → Validate / Block / Log → Continue or Abort.
Toàn bộ flow tự động — agent không thể bypass nếu không vô hiệu hóa hook trong settings.json.

Kỹ thuật 04
Memory System — 3-Tier Architecture
Context là tài nguyên quý nhất. Memory phân tầng giúp tối ưu token usage mà vẫn đảm bảo đúng thông tin đúng lúc.
Tier 1 — Always Loaded (mọi session) HARD LIMIT: ≤ 50 lines
MEMORY.md — Master Index → Pointer đến Tier 2 files · Auto-trigger /dream khi >40 lines
Tier 2 — On-Demand (max 3 files/session, load bằng Relevance Gate) 3-Question Gate
annotations.md project_tech_decisions.md user_role.md feedback_rules.md reference_links.md gitnexus-registry.md
Tier 3 — Search-Only (KHÔNG bao giờ load chủ động) Grep only khi user yêu cầu
archive/sessions/ archive/decisions/ archive/dreams/
3-Question Relevance Gate — phải qua cả 3 mới được load Tier 2
Q1 Task này CÓ THỰC SỰ cần nội dung file này? Keyword match ≠ load. Nếu không nói được dùng fact nào → skip.
Q2 Tôi sẽ dùng nó trong 3 agent turns tới không? "Có thể sau" = KHÔNG load.
Q3 Một subsection (line range) có đủ không? Nếu có → đọc từng đoạn thay vì cả file.

Kỹ thuật 05
Auto-Dream — Memory Consolidation
5-phase reflective cleanup tự động ngăn memory bloat và dữ liệu stale tích lũy qua nhiều session.
5 Execution Phases
1
Orient
Đánh giá sức khỏe memory directory, đọc MEMORY.md, kiểm tra topic files hiện có, xác định line count
2
Detect
Tìm file stale, entries sai, broken links trong MEMORY.md, duplicated memories, drifted facts
3
Consolidate
Merge memories trùng lặp, update stale facts, tạo Tier 2 files mới nếu có topic chưa được ghi nhận
4
Prune
Xóa orphaned MEMORY.md entries, sửa YAML frontmatter violations, trim MEMORY.md về <40 lines
5
Archive
Di chuyển file cũ → archive/decisions/; ghi log actions vào archive/dreams/
3 Auto-trigger Conditions
📏 MEMORY.md vượt 40 lines (tiệm cận hard limit 50)
🔢 Mỗi session thứ 5 — maintenance định kỳ
🕰️ 3+ Tier 2 files không truy cập 7+ ngày
Invocation Paths
🔄 Auto: session-stop.sh → conditions met → trigger
💬 Manual: /dream slash command
📋 Overflow: MEMORY.md lint fail
Fail-open rule: Nếu auto-dream fail/timeout → log vào active.md và tiếp tục session. Không bao giờ block foreground task.
Kỹ thuật 06
Agent Audit Trail — Structured Logging
Mọi subagent invocation được ghi lại tự động — machine-queryable JSONL và human-readable log song song.
JSONL Format (agent-audit.jsonl)
// production/session-logs/
{
  "event":      "SubagentStart",
  "timestamp":  "2026-04-15T10:30:00Z",
  "session_id": "session-abc123",
  "agent_id":   "agent-xyz789",
  "agent_name": "backend-developer",
  "branch":     "main"
}
Text Format (agent-audit.log)
2026-04-15T10:30:00Z | backend-developer (agent-xyz) | branch: main | session: abc123
Log Files
FileFormatDùng để
agent-audit.jsonl JSONL jq query, analytics, audit
agent-audit.log Text Human quick scan
session-log.md Markdown Session stats tổng hợp
Guarantee: Hook SubagentStart bắt mọi invocation qua Agent tool. Không agent nào spawn được mà không để lại audit record.
Queries có thể chạy
# Đếm số lần mỗi agent được dùng
jq -r '.agent_name' audit.jsonl | sort | uniq -c

# Agent nào dùng nhiều nhất hôm nay
jq -r 'select(.timestamp | startswith("2026-04-15"))' \
  audit.jsonl | jq -r '.agent_name'

Multi-Agent Orchestration — Wave Execution
Skill /orchestrate phân tích dependency, xây execution waves từ 27 agent domains, và trình bày kế hoạch cho user phê duyệt trước khi chạy.
Execution Phases
1
Ground Phase
Đọc CLAUDE.md, agent-roster.md, PRD, Architecture, technical decisions — xây context đầy đủ trước khi plan
2
Decompose & Route
Map task → 27 agent domains; xác định specialist cần thiết dựa trên domain routing table
3
Dependency Analysis
tech-director → all · data-engineer → backend · ux-designer → frontend · backend → frontend
4
Wave Planning → User Approval
Present plan đầy đủ; KHÔNG execute cho đến khi được approve
5
Wave Execution
Agents trong cùng wave chạy parallel; waves chạy sequential theo dependency order
6
Synthesize
Combine outputs từ tất cả agents → unified result cho user

Kỹ thuật 08
Fork-Join — Parallel Worktree Execution
Git worktree isolation cho phép N agents làm việc hoàn toàn độc lập trên cùng codebase — không conflict, không chờ đợi.
CLI Commands (fork-join.sh)
CommandAction
fork <branch> <dir> [base] Tạo git worktree riêng biệt từ base branch
status <dir> Xem commits, staged files, modified files
list Liệt kê tất cả active worktrees
join <dir> [--no-delete] Merge branch về main, xóa worktree
purge <dir> Force remove không merge (discard)
Execution Workflow
1
fork → N isolated worktrees
Mỗi unit có branch riêng, thư mục riêng
2
N subagents chạy đồng thời
Mỗi agent trong worktree của mình — zero conflict
3
status → verify completion
Kiểm tra từng worktree trước khi join
4
join → merge + cleanup
Sequential merge về main; uncommitted changes block join
Parallel-Safe Decision Rule
✕
Cùng file
Hai agents cần write cùng 1 file → Sequential bắt buộc
✕
Output dependency
Agent B cần output của Agent A → Sequential bắt buộc
✓
Không liên hệ
Khác file, khác domain, không phụ thuộc nhau → Parallel-safe → Fork!
Fallback: Nếu <2 units parallelizable → suggest /orchestrate thay thế Fork-Join.
Worktree Isolation (Coordination Rule #10)
Dùng isolation: "worktree" cho bất kỳ task nào có nguy cơ break working code: large refactors (5+ files), experimental approaches, DB migrations.

Kỹ thuật 13
Safety Guard — bash-guard.sh
2-level protection chạy tự động trước mọi Bash command — Hard Block không thể bypass, Soft Warn yêu cầu xác nhận.
🛑 HARD BLOCK — Abort Immediately
:(){ :|:& };: Fork bomb — crash toàn bộ shell process tree
mkfs.* Format disk/partition — không thể undo
dd if=/dev/zero Overwrite data với zeros liên tục
crontab -r Xóa toàn bộ cron jobs không có backup
rm -rf / Xóa root filesystem
⚠ SOFT WARN — Ask Confirmation
DROP TABLE Xóa database table — cần confirm schema backup
DROP DATABASE Xóa toàn bộ database — irreversible trong prod
git reset --hard Xóa uncommitted changes không thể recover
rm -rf /path/... Xóa thư mục lớn — kiểm tra path trước
git push --force main Force push lên protected branch
git push --force develop Overwrite shared branch history
Risk Tier Assessment
Low Reversible · local only · no shared state
Medium Shared code · cần rollback plan
High Destructive · prod impact · explicit approval required
Nguyên tắc: Khi gặp obstacle, không dùng destructive actions để bypass. Tìm root cause và fix underlying issue. Chỉ dùng --no-verify, reset --hard khi user explicitly yêu cầu.

✦ Hooks chạy tự động — không cần user trigger
✦ Memory bounded — auto-consolidate khi cần
✦ State persisted — recover sau mọi loại crash
✦ Quality enforced — commit không thể bypass

Core Design Philosophy
File là bộ nhớ — không phải conversation.
Hooks là safety net — không phải afterthought.

Memory là tiered — không phải flat dump.
